# REQUIREMENTS.md — のんだっけ？（服薬・摂取記録カレンダーアプリ）

## 1. プロジェクト概要

### コンセプト
「あれ、いつ飲んだっけ？」を解消する、**記録して振り返る**カレンダーアプリ。アプリ名「のんだっけ？」はこの課題そのもの。
薬に限定せず、**食事以外で口に入れるもの**（薬・サプリ・その他）をまとめて管理する。

### ポジショニング
- 既存の服薬管理アプリ（お薬手帳系・リマインダー系）との差別化点は **「N日に1回」「週◯回」などの周期服用への特化** と **医療色の薄さ**
- リマインダーアプリではなく「摂取記録カレンダー」。通知は主機能にしない（MVPでは非搭載）

### ターゲット
- 一次：開発者自身（3日に1回・食後服用の管理）
- 二次：周期服用のある一般ユーザー、シニア層（1日おき・週1服薬が多い）
- 配布：Google Play で一般公開（完全無料）

## 2. 技術スタック

| 項目 | 選定 |
|------|------|
| フレームワーク | Expo (React Native) + TypeScript |
| データ保存 | expo-sqlite（ローカル完結・クラウドなし） |
| ビルド/配布 | EAS Build → Google Play（Android先行） |
| バックエンド | なし（外部API・認証・課金SDKすべて不要） |
| 通知 | なし（Phase 2候補） |

- 端末内完結のため、要求権限は最小限にする（ネットワーク権限も原則不要）
- iOS対応はコード上クロスプラットフォームを維持するが、ストア申請はAndroid成功後に判断

## 3. データモデル

### items（登録アイテム）
| カラム | 型 | 説明 |
|--------|-----|------|
| id | INTEGER PK | |
| name | TEXT | アイテム名 |
| category | TEXT | `medicine` / `supplement` / `other`（固定3種） |
| schedule_type | TEXT | `daily` / `interval` / `weekly` / `as_needed` |
| interval_days | INTEGER? | schedule_type=interval のとき（例：3日に1回→3） |
| weekdays | TEXT? | schedule_type=weekly のとき（例：`"1,4"`＝月・木） |
| timings | TEXT | 服用タイミングのCSV：`morning` / `noon` / `evening` / `bedtime`（複数可） |
| memo | TEXT? | 用量・注意事項など自由記入 |
| is_active | INTEGER | 服用中フラグ（中止しても履歴は残す） |
| created_at | TEXT | |

### intake_records（摂取記録）
| カラム | 型 | 説明 |
|--------|-----|------|
| id | INTEGER PK | |
| item_id | INTEGER FK | |
| taken_date | TEXT | 服用日（YYYY-MM-DD）。過去日の追記・修正が前提のため recorded_at と分離 |
| timing | TEXT | `morning` / `noon` / `evening` / `bedtime` / `none`（1日1回系はnone可） |
| recorded_at | TEXT | 実際に記録した日時 |

- UNIQUE制約：(item_id, taken_date, timing) — 二重記録防止
- 「毎日」は interval_days=1 に集約してもよい（実装judgment）

## 4. デザイン方針

方向性：**医療アプリ然としない、シンプルでモダンなUI**（参考：サブスクBoxのストアスクリーンショット）

- 白ベース＋アクセントカラー1色（ミント/ティール系を第一候補。医療系定番の青・赤は避ける）
- カレンダーの日セルに、登録アイテムの**丸アイコン**（カテゴリ色 or アイテムごとの絵文字/頭文字）を表示し、パッと見で「飲む日」がわかる
- カレンダー上部に**サマリーカード**を1枚：「今日のむもの ◯件（済 ◯／残り ◯）」など、大きな数字1つで状態が伝わるデザイン
- カテゴリ切替はカレンダー上部のタブ（チップ型セグメント）で表示・操作
- 「飲んだ」記録ボタンは、絵文字＋大きめの角丸カードUI（例：✅「のんだ！」）。タップした瞬間の気持ちよさを重視
- 角丸・余白多め・影は控えめ。装飾やグラデーションは使わない
- 下部タブナビ3つ：**カレンダー（ホーム）／一覧／設定**
- タップ領域は最低44dp以上を維持（シニア特化デザインにはしないが、結果的に誰にでも扱いやすいシンプルさを目指す）

## 5. 画面構成

### 5.1 ホーム＝カレンダー画面（起動時に直接表示）
- 月表示カレンダー
- **タブでカテゴリ切替**：「全部」／「お薬」／「サプリ」／「その他」を画面上部のタブで選択
  - 当初は左右スワイプ＋ドットインジケーターだったが、視認性の問題からタブUIに変更（2026-07）
- 日セルの表示：
  - 服用予定あり：カテゴリ色のマーカー
  - 記録済み：チェック等の完了表示
  - 予定日なのに未記録（過去日）：未記録とわかる表示（グレー/枠のみ等）
- 日セルタップ → 日別詳細（下からシート表示）

### 5.2 日別詳細シート
- その日の予定アイテム＋記録済みアイテムを一覧表示
- タイミング（朝・昼・夕・就寝前）ごとにワンタップで「飲んだ」記録
- **過去日の追記・修正・削除が可能**（飲み忘れに後から気づくケースが主要ユースケース）
- 不定期（as_needed）アイテムはこの画面から任意日に記録追加できる

### 5.3 アイテム一覧画面
- 登録済みアイテムをカテゴリ別に一覧表示（「何を飲んでいるか」の整理・棚卸しが目的）
- 各アイテム：名前／カテゴリ／服用パターン／最終服用日を表示
- タップで編集、服用中止（is_active=false）操作

### 5.4 アイテム登録・編集画面
- **必須項目は「名前」と「服用パターン」の2つのみ**。カテゴリはデフォルト「お薬」、タイミング・メモは任意
- 名前は正式な薬名でなくてよい（本アプリはお薬手帳ではなく記録アプリ）。プレースホルダーで明示：「例：血圧の薬、ビタミンD など自分がわかる名前でOK」
- 薬名データベース・検索機能は持たない（「登録されている薬がない」問題を構造的に回避）
- 30秒で登録完了できることを目標とする（競合の低評価要因＝入力項目過多への対策）

### 5.5 設定画面
- データ全削除、アプリ情報、プライバシーポリシー表示（ストア公開要件）

## 6. 機能要件（MVP）

1. アイテムのCRUD（固定3カテゴリ、4種の服用パターン、朝昼夕就寝前タイミング）
2. カレンダー表示（月）＋タブによるカテゴリ切替
3. 「飲んだ」記録のワンタップ登録（当日・過去日とも）、修正・削除
4. 予定日の自動算出 — **予定日はDBに保存せず、常に記録とスケジュール設定から導出する**
   - interval型：次回予定＝最終服用日＋N日（自動でずれる方式）。記録が1件もない場合は開始日を初回予定とする
   - 予定日を過ぎて未服用の場合、予定は消えず「今日のむもの」に期限超過として残り続ける。服用記録が入った日を起点に次回を再計算
   - daily型・weekly型：カレンダー固定（飲み忘れてもずれない）
   - 過去日の追記・修正・削除時も、導出方式のため予定は自動で再計算される（予定の整合性を保つ特別な処理は不要）
5. アイテム一覧（棚卸しビュー）
6. データはすべて端末内SQLiteに保存

## 7. 非機能要件

- 通知・バックグラウンド処理なし
- ネットワーク通信なし（プライバシー訴求点としてストア説明文に明記）
- 完全無料・広告なし・課金SDKなし
- UIはシンプルさ最優先（デザイン方針参照）。タップ領域は最低44dp
- Google Play公開に必要な事項（プライバシーポリシー、データセーフティ申告＝収集なし）

## 8. Phase 2 候補（本フェーズでは実装しない・詳細記述も省略）

- 控えめな通知（服用予定日の朝1回など）
- クラウド同期・機種変更時のデータ移行（エクスポート/インポート）
- iOS版ストア申請
- 服用率などの振り返り統計
- 写真からのアイテム名読み取り（Gemini Vision等）※「ネットワーク通信なし」方針と衝突するため、実装する場合もオプトインのオプション機能とする

- 薬剤データベース連携・薬の検索
- お薬手帳機能・医療機関連携
- 家族共有・複数ユーザー
- 飲み合わせチェック等の医療判断に関わる機能（アプリは記録ツールであり医療アドバイスを行わない旨をストア説明・アプリ内に明記）
